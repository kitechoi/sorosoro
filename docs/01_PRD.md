# 01. PRD — 소로소로(SOROSORO)

> 하루하루 재봉 과정을 기록하는 개인 재봉일기 서비스 (SewingLog SOROSORO)

---

# 1. 문서 개요

## 1.1 문서 목적

본 문서는 **소로소로(SOROSORO)** 서비스의 제품 기획 방향을 정의하는 PRD(Product Requirements Document)이다.

이 문서는 다음 내용을 설명한다.

- 왜 이 서비스를 만드는가
    
- 어떤 사용자의 문제를 해결하는가
    
- 어떤 기능을 제공하는가
    
- MVP에서 무엇을 구현하고, 무엇을 제외하는가
    
- 향후 어떤 방향으로 확장할 수 있는가
    

구체적인 API, ERD, 백엔드 구현 방식, 트랜잭션 경계, 패키지 구조 등은 별도 문서인 SRS, ADR, Backend Design에서 다룬다.

---

# 2. 서비스 개요

## 2.1 서비스명

**소로소로(SOROSORO)**

## 2.2 한 줄 소개

하루하루 재봉 과정을 기록하고, 작품과 원단을 연결해 관리하는 개인 재봉일기 서비스

## 2.3 서비스 설명

소로소로는 재봉을 취미로 하는 사용자가 작품을 만드는 과정을 날짜별로 기록하고, 구매한 원단을 체계적으로 관리할 수 있도록 돕는 개인 기록 서비스이다.

재봉 작업은 하나의 작품을 완성하기까지 여러 날에 걸쳐 진행된다. 패턴을 그리고, 원단을 재단하고, 심지를 붙이고, 재봉하고, 수정하고, 마감하는 과정이 모두 나뉘어 있다. 그러나 대부분의 기록은 사진첩, 메모 앱, SNS, 쇼핑몰 주문 내역 등에 흩어져 남는다.

소로소로는 이러한 기록을 **Project**, **DailyLog**, **Fabric**이라는 구조로 정리한다.

- **Project**는 하나의 작품을 의미한다.
    
- **DailyLog**는 특정 날짜에 특정 프로젝트에서 진행한 작업 기록을 의미한다.
    
- **Fabric**은 사용자가 구매한 원단 기록을 의미한다.
    

이를 통해 사용자는 단순히 완성 사진만 남기는 것이 아니라, 작품이 완성되어 가는 과정과 사용한 원단 이력을 함께 관리할 수 있다.

---

# 3. 문제 정의

## 3.1 작업 과정이 흩어져 남는다

재봉은 여러 날에 걸쳐 진행되는 과정 중심의 취미이다.

예를 들어 카드지갑 하나를 만들더라도 다음과 같은 작업이 발생한다.

- 패턴 그리기
    
- 원단 재단
    
- 심지 부착
    
- 지퍼 달기
    
- 안감 연결
    
- 겉감 봉제
    
- 마감 및 수정
    

하지만 사용자는 보통 사진첩에 중간 사진을 저장하거나, 메모 앱에 간단한 기록을 남기거나, 완성 후 SNS에 사진을 올리는 정도로 기록한다.

이 경우 시간이 지나면 다음과 같은 질문에 답하기 어렵다.

- 이 작품은 언제 시작했지?
    
- 지난번에 어디까지 작업했지?
    
- 이 부분은 왜 수정했지?
    
- 완성까지 총 얼마나 걸렸지?
    

---

## 3.2 작품과 원단의 관계를 관리하기 어렵다

재봉에서는 프로젝트를 먼저 정하고 원단을 구매하는 경우도 있지만, 반대로 마음에 드는 원단을 먼저 구매한 뒤 나중에 어울리는 작품을 만드는 경우도 많다.

즉, 원단은 프로젝트의 하위 데이터가 아니라 사용자가 독립적으로 관리하는 자산에 가깝다.

하지만 일반적인 메모 앱이나 사진첩에서는 다음 관계를 관리하기 어렵다.

- 이 프로젝트에 어떤 원단을 사용했는가
    
- 이 원단으로 어떤 작품들을 만들었는가
    
- 같은 원단을 다시 사용한 적이 있는가
    
- 다시 구매할 만한 원단이었는가
    

소로소로는 Project와 Fabric을 독립적으로 관리하면서, 둘을 연결할 수 있도록 설계한다.

---

## 3.3 원단 구매 정보를 다시 찾기 어렵다

원단을 구매할 때는 상품명, 품번, 구매처, 가격, 색상, 원단폭, 혼용률 등을 확인하지만, 시간이 지나면 이 정보가 쇼핑몰 주문 내역이나 상품 상세페이지에 흩어진다.

특히 여러 쇼핑몰에서 원단을 구매하는 경우, 사용자는 나중에 다음 정보를 다시 찾기 어렵다.

- 어디서 샀는지
    
- 얼마에 샀는지
    
- 어떤 색상이었는지
    
- 품번이 무엇이었는지
    
- 원단폭과 혼용률이 무엇이었는지
    
- 재구매할 만했는지
    

결국 사용자는 같은 정보를 반복해서 검색하거나, 이전 구매 이력을 제대로 활용하지 못한다.

---

## 3.4 원단 정보 입력이 번거롭다

원단 쇼핑몰마다 상품 정보의 위치와 표현 방식이 다르다.

어떤 사이트는 상단에 상품명과 옵션이 잘 정리되어 있지만, 어떤 사이트는 상세페이지를 내려야 혼용률과 원단폭을 확인할 수 있다. 또 어떤 사이트는 이미지 안에만 정보가 들어 있기도 하다.

사용자가 직접 기록해야 하는 정보는 다음과 같다.

- 상품명
    
- 품번
    
- 색상
    
- 구매처
    
- 구매 가격
    
- 원단 규격
    
- 원단폭
    
- 혼용률
    
- 상품 URL
    

이 입력 과정이 번거로우면 사용자는 기록을 미루거나 포기하게 된다.

---

# 4. 제품 목표

## 4.1 기록 경험을 단순화한다

사용자가 재봉 작업 후 빠르게 일지를 작성할 수 있도록 한다.

핵심은 긴 글을 쓰는 것이 아니라, 다음 정보를 간단히 남기는 것이다.

- 오늘 어떤 프로젝트를 작업했는가
    
- 어떤 작업을 했는가
    
- 몇 시부터 몇 시까지 작업했는가
    
- 어떤 사진을 남길 것인가
    
- 다음에 기억해야 할 메모는 무엇인가
    

---

## 4.2 작품의 제작 과정을 되돌아볼 수 있게 한다

Project는 단순한 작품 카드가 아니라, 여러 DailyLog가 모여 만들어지는 결과 보고서 역할을 한다.

프로젝트 상세에서는 다음을 확인할 수 있어야 한다.

- 프로젝트 기본 정보
    
- 프로젝트 진행 상태
    
- 전체 작업 기간
    
- 총 작업 시간
    
- 날짜별 작업 기록
    
- 사용한 원단
    
- 참고한 패턴 또는 URL
    
- 회고
    

---

## 4.3 원단을 독립적으로 관리한다

Fabric은 프로젝트에 종속되지 않고 독립적으로 존재한다.

사용자는 원단을 먼저 등록해두고, 나중에 프로젝트에 연결할 수 있어야 한다.

Fabric에서는 다음을 확인할 수 있어야 한다.

- 원단의 구매 정보
    
- 상품 정보
    
- 원단 사진
    
- 재구매 의사
    
- 사용된 프로젝트 목록
    

---

## 4.4 작업 습관을 시각화한다

사용자는 재봉 작업을 얼마나 자주, 얼마나 오래 했는지 확인할 수 있어야 한다.

이를 위해 다음 시각화 기능을 제공한다.

- 월별 캘린더
    
- 날짜별 작업 기록
    
- 작업 시간 기반 재봉 잔디
    

재봉 잔디는 GitHub Contributions와 유사한 형태를 참고하되, 커밋 횟수가 아니라 **작업 시간**을 기준으로 색상을 표현한다.

---

## 4.5 이미지 중심 UX를 빠르게 제공한다

재봉 기록은 사진의 비중이 높다.

따라서 이미지를 느리게 불러오면 서비스의 사용성이 크게 떨어진다.

소로소로는 이미지 업로드와 조회를 서비스 핵심 기술 과제로 보고, 다음 방향을 지향한다.

- S3 Presigned URL 기반 직접 업로드
    
- 원본, Medium, Thumbnail 이미지 분리
    
- 비동기 이미지 리사이징
    
- CloudFront를 통한 빠른 이미지 조회
    

---

# 5. 타깃 사용자

## 5.1 1차 타깃 사용자

재봉을 취미로 하며, 자신의 작업 과정을 기록하고 싶은 개인 사용자

특징

- 원단을 여러 쇼핑몰에서 구매한다.
    
- 하나의 작품을 여러 날에 걸쳐 만든다.
    
- 중간 과정 사진을 자주 찍는다.
    
- 완성 후 어떤 원단을 썼는지 다시 찾고 싶어 한다.
    
- 재구매할 만한 원단인지 기록하고 싶어 한다.
    

## 5.2 사용 환경

MVP는 **모바일 웹 중심**으로 설계한다.

- 사용자는 작업 중간에 휴대폰으로 사진을 찍고 기록할 가능성이 높다.
    
- 재봉 작업 공간에서 PC보다 스마트폰 접근성이 높다.
    
- PC에서도 사용할 수 있지만 핵심 UX는 모바일을 기준으로 설계한다.
    

---

# 6. 핵심 도메인

## 6.1 Project

Project는 하나의 작품을 의미한다.

예시

- 딸기 리본 카드지갑
    
- 린넨 파자마 바지
    
- 플라워 파우치
    
- 스트라이프 에코백
    

Project는 다음 정보를 가진다.

- 프로젝트명
    
- 설명
    
- 진행 상태
    
- 시작일
    
- 종료일
    
- 회고
    
- 대표 이미지
    
- 프로젝트 설정
    
- 사용 원단
    
- 작업 일지
    

---

## 6.2 DailyLog

DailyLog는 특정 날짜에 특정 프로젝트에서 수행한 작업 기록이다.

하루에 여러 프로젝트를 작업할 수 있으므로, 하루에 여러 DailyLog가 작성될 수 있다.

예시

- 2026-06-27 / 파자마 바지 / 패턴 그리기, 재단
    
- 2026-06-27 / 카드지갑 / 지퍼 달기, 단추 달기
    

DailyLog는 다음 정보를 가진다.

- 작업 날짜
    
- 프로젝트
    
- 작업 종류
    
- 작업 시간 구간
    
- 총 작업 시간
    
- 사진
    
- 메모
    

---

## 6.3 Fabric

Fabric은 사용자가 구매한 원단 기록이다.

Fabric은 Project에 종속되지 않는다. 사용자는 원단만 먼저 등록할 수 있고, 나중에 여러 프로젝트에 연결할 수 있다.

Fabric은 다음 정보를 가진다.

- 사용자가 부르는 원단명
    
- 쇼핑몰 상품명
    
- 품번
    
- 상품 URL
    
- 구매처
    
- 구매일
    
- 구매 가격
    
- 색상
    
- 규격
    
- 원단폭
    
- 혼용률
    
- 메모
    
- 평점
    
- 재구매 의사
    
- 사진
    

---

## 6.4 ProjectFabric

ProjectFabric은 Project와 Fabric의 연결을 의미한다.

단순한 조인 테이블이 아니라, 특정 프로젝트에서 해당 원단을 어떤 맥락으로 사용했는지 기록할 수 있는 연결 엔티티이다.

예시

- 안감으로 사용
    
- 리본 포인트 부분에 사용
    
- 생각보다 두꺼워서 일부에만 사용
    

---

## 6.5 Photo

Photo는 Project, DailyLog, Fabric에서 공통으로 사용하는 이미지 엔티티이다.

사진 정책은 다음과 같다.

- Project: 최대 15장, 대표 이미지 1장
    
- DailyLog: 최대 15장, 대표 이미지 1장
    
- Fabric: 최대 2장
    

---

# 7. 주요 기능

## 7.1 카카오 로그인

MVP에서는 카카오 소셜 로그인만 제공한다.

사용자는 별도의 이메일 회원가입 없이 카카오 계정으로 서비스를 시작할 수 있다.

인증 후 서버는 Access Token과 Refresh Token을 발급한다.

---

## 7.2 프로젝트 관리

사용자는 작품 단위의 프로젝트를 생성하고 관리할 수 있다.

주요 기능

- 프로젝트 생성
    
- 프로젝트 목록 조회
    
- 프로젝트 상세 조회
    
- 프로젝트 수정
    
- 프로젝트 삭제
    
- 프로젝트 상태 관리
    
- 프로젝트 회고 작성
    

프로젝트 상태는 다음과 같다.

- IN_PROGRESS
    
- ON_HOLD
    
- COMPLETED
    
- ARCHIVED
    

---

## 7.3 프로젝트 설정 관리

프로젝트에는 제작 과정에서 자주 변하지 않는 설정값이 존재한다.

예시

- 패턴 방식
    
- 패턴명
    
- 완성 가로 길이
    
- 완성 세로 길이
    
- 완성 규격 메모
    
- 재봉틀 세팅
    
- 사용 바늘
    
- 사용 실
    
- 참고 URL
    

이 정보는 DailyLog가 아니라 ProjectSpecification으로 관리한다.

패턴 방식은 다음 세 가지로 구분한다.

- SELF_DRAFTED: 처음부터 직접 만든 패턴
    
- COPIED: 기존 패턴을 그대로 사용
    
- MODIFIED: 기존 패턴을 참고하되 치수, 구조, 디테일을 변형
    

---

## 7.4 재봉 일기 작성

사용자는 프로젝트별로 날짜별 작업 일지를 작성할 수 있다.

주요 기능

- 작업 날짜 입력
    
- 작업 종류 선택
    
- 여러 작업 시간 구간 입력
    
- 작업 사진 업로드
    
- 메모 작성
    

작업 종류는 Enum으로 관리한다.

예시

- PATTERN_DRAFTING
    
- CUTTING
    
- SEWING
    
- PRESSING
    
- FITTING
    
- FIXING
    
- FINISHING
    
- ETC
    

DailyLog의 총 작업 시간은 사용자가 직접 입력하지 않고, TimeEntry의 합산값으로 계산한다.

---

## 7.5 원단 관리

사용자는 구매한 원단을 독립적으로 등록하고 관리할 수 있다.

주요 기능

- 원단 등록
    
- 원단 목록 조회
    
- 원단 상세 조회
    
- 원단 수정
    
- 원단 삭제
    
- 원단 사진 등록
    
- 재구매 의사 기록
    

---

## 7.6 프로젝트-원단 연결

사용자는 프로젝트에 원단을 연결할 수 있다.

이를 통해 다음 흐름이 가능하다.

- Project 상세 → 사용한 원단 조회
    
- Fabric 상세 → 이 원단이 사용된 프로젝트 조회
    

동일 프로젝트에 동일 원단은 중복 연결할 수 없다.

---

## 7.7 캘린더

사용자는 월별 캘린더에서 날짜별 작업 여부와 작업 시간을 확인할 수 있다.

캘린더에서 특정 날짜를 선택하면 해당 날짜에 작성한 DailyLog 목록을 볼 수 있다.

---

## 7.8 재봉 잔디

재봉 잔디는 연도별 작업 시간을 시각화하는 기능이다.

기준은 작업 횟수가 아니라 하루 총 작업 시간이다.

예시 기준

- 0분: Level 0
    
- 1~30분: Level 1
    
- 31~120분: Level 2
    
- 121~240분: Level 3
    
- 241분 이상: Level 4
    

---

## 7.9 이미지 업로드 및 최적화

사용자는 Project, DailyLog, Fabric에 사진을 업로드할 수 있다.

이미지는 다음 형태로 관리한다.

- Original
    
- Medium
    
- Thumbnail
    

업로드는 S3 Presigned URL을 사용하고, 리사이징은 Queue 기반 Worker가 비동기로 처리한다.

---

## 7.10 AI 기반 원단 정보 자동 기록

본 기능은 2차 기능으로 정의한다.

사용자가 쇼핑몰 주문 내역 캡처 이미지를 업로드하면, AI가 상품명, 색상, 옵션 등을 추출한다.

이후 상품 검색 및 상세 정보 수집을 통해 원단 정보를 자동 입력하고, 사용자가 최종 확인 후 Fabric으로 저장할 수 있도록 한다.

MVP에서는 본 기능을 구현하지 않지만, 향후 확장을 고려하여 별도 Job 기반 구조로 설계한다.

# 10. 기대 효과

소로소로는 단순한 작품 저장 서비스가 아니라, 프로젝트, 작업 과정, 원단 구매 이력을 하나의 데이터로 연결하여 사용자가 시간이 지나도 자신의 재봉 기록을 다시 찾아볼 수 있는 **개인 재봉 기록 플랫폼**을 목표로 한다.

## 10.1 작업 과정의 누적

사용자는 완성 사진만 저장하는 것이 아니라, 작품이 완성되어 가는 과정을 날짜별로 남길 수 있다.

이를 통해 시간이 지나도 다음 내용을 쉽게 확인할 수 있다.

- 어떤 과정을 거쳐 작품을 완성했는지
    
- 어느 날짜에 어떤 작업을 했는지
    
- 작업 중 어떤 시행착오가 있었는지
    
- 완성까지 총 얼마나 걸렸는지
    

---

## 10.2 원단 이력의 재활용

사용자는 구매한 원단 정보를 별도로 관리하고, 이를 프로젝트와 연결할 수 있다.

이를 통해 다음 내용을 쉽게 확인할 수 있다.

- 어떤 작품에 어떤 원단을 사용했는지
    
- 특정 원단으로 어떤 작품들을 만들었는지
    
- 어떤 원단을 다시 구매하고 싶은지
    
- 이전 구매 가격과 구매처가 무엇이었는지
    

---

## 10.3 기록 습관의 시각화

사용자는 캘린더와 재봉 잔디를 통해 자신의 작업 습관을 시각적으로 확인할 수 있다.

이를 통해 단순히 결과물을 모으는 것을 넘어, 꾸준히 만드는 과정 자체를 기록하고 동기부여를 얻을 수 있다.

---

## 10.4 포트폴리오 관점의 효과

본 프로젝트는 단순 CRUD 서비스가 아니라, 다음 설계 역량을 보여주는 포트폴리오 산출물이 될 수 있다.

- Project와 Fabric을 독립 도메인으로 분리한 모델링
    
- DailyLog와 DailyLogTimeEntry를 분리한 작업 시간 관리 구조
    
- ProjectFabric을 연결 엔티티로 설계한 이유
    
- Photo 공통 엔티티 설계
    
- S3 Presigned URL 기반 이미지 업로드 구조
    
- Queue + Worker 기반 비동기 이미지 리사이징 구조
    
- 향후 AI 자동 기록 기능까지 고려한 확장 가능성

# 11. 성공 기준

MVP의 성공 기준은 다음과 같다.

## 11.1 기능적 성공 기준

- 사용자는 카카오 로그인을 통해 서비스를 이용할 수 있다.
    
- 사용자는 프로젝트를 생성하고 작업 일지를 작성할 수 있다.
    
- 사용자는 원단을 등록하고 프로젝트와 연결할 수 있다.
    
- 사용자는 날짜별 작업 기록을 캘린더에서 확인할 수 있다.
    
- 사용자는 작업 시간 기반 재봉 잔디를 확인할 수 있다.
    
- 사용자는 이미지를 업로드하고 빠르게 조회할 수 있다.
    

## 11.2 포트폴리오 성공 기준

- 단순 CRUD가 아니라 도메인 모델링이 드러난다.
    
- Project와 Fabric을 독립 도메인으로 설계한 이유가 설명된다.
    
- DailyLogTimeEntry를 분리한 이유가 설명된다.
    
- 이미지 업로드와 리사이징 파이프라인이 설명된다.
    
- PostgreSQL, S3, Queue, Worker 선택 이유가 설명된다.
    
- 실제 서비스를 개발하기 전 수준의 설계 문서가 존재한다.

# 12. 향후 확장 방향

## 12.1 AI 원단 자동 기록

주문 내역 캡처 이미지를 기반으로 원단 정보를 자동 추출한다.

예상 흐름

1. 캡처 이미지 업로드
    
2. Extraction Job 생성
    
3. AI가 상품명, 색상, 옵션 추출
    
4. 상품 상세 정보 검색
    
5. 사용자 검수
    
6. Fabric 생성
    

---

## 12.2 커뮤니티

사용자가 자신의 프로젝트를 공개하고 다른 사용자와 공유할 수 있도록 한다.

단, 커뮤니티는 MVP 범위에서 제외한다.

향후 커뮤니티 기능 도입 시 Project를 그대로 공개하지 않고, 별도의 CommunityPost 또는 Snapshot 구조를 고려한다.

---

## 12.3 원단 가격 비교

Fabric에 저장된 상품 URL을 기반으로 현재 가격을 다시 확인하고, 구매 당시 가격과 비교할 수 있다.

---

## 12.4 통계 대시보드

사용자의 재봉 활동을 통계로 제공한다.

예시

- 월별 작업 시간
    
- 가장 많이 한 작업 종류
    
- 가장 많이 사용한 원단
    
- 완성 프로젝트 수
    
- 재구매 의사 YES 비율
    

---

# 13. 정리

소로소로는 재봉을 취미로 하는 사용자가 작품 제작 과정과 원단 구매 이력을 함께 기록할 수 있도록 돕는 개인 재봉일기 서비스이다.

핵심은 단순히 완성 사진을 저장하는 것이 아니라, 작품이 완성되어 가는 과정을 날짜별로 기록하고, 사용한 원단과 연결하여 시간이 지나도 다시 찾아볼 수 있는 구조를 만드는 것이다.

MVP에서는 기록 경험, 원단 연결, 이미지 최적화에 집중하고, AI 기반 자동 기록과 커뮤니티 기능은 향후 확장 기능으로 분리한다.
